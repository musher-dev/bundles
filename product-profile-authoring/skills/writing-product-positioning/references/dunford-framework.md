# Dunford 5-Component Positioning Framework

Based on April Dunford's "Obviously Awesome" methodology. The five components must be worked through sequentially — each builds on the output of the previous one.

## The Sequential Flow

```
Competitive       Differentiated       Customer       Target Customer       Market
Alternatives  →   Capabilities    →    Value     →      Segments       →   Category
     1                 2                 3                  4                 5

"What would      "What can we       "So what?       "Who cares        "What context
 they use          do that they      What does        most about         makes our
 instead?"         can't?"           that enable?"    this value?"       value obvious?"
```

**Why the order matters**: Starting with your own features leads to inside-out positioning ("we're great because..."). Starting with alternatives leads to outside-in positioning ("given what exists, here's why we matter").

## Component Definitions

### 1. Competitive Alternatives

**Question**: What would customers do if your product didn't exist?

**Types of alternatives:**
| Type | Definition | Example |
|---|---|---|
| Direct competitor | Product in the same category solving the same problem | Competitor X's product |
| Indirect competitor | Different approach to the same problem | Using a general-purpose tool (e.g., Excel) |
| Status quo | The current process without any dedicated tool | Manual process, email, spreadsheets, "do nothing" |

**Rules:**
- Always include at least one status quo alternative
- List 3-5 alternatives total
- For each, identify the key limitation that creates an opening for your product

### 2. Differentiated Capabilities

**Question**: What can your product do that the alternatives from Component 1 cannot?

**Rules:**
- Each capability must trace back to a specific alternative's limitation
- Must be verifiable — include proof (demo, benchmark, open source code, customer testimony)
- "Better" is not a capability — be specific about what is different
- 3-5 capabilities maximum; if you have more, some are features, not differentiators

**Quality test**: If a competitor could say the same thing, it's not differentiated.

### 3. Customer Value

**Question**: What value do the differentiated capabilities from Component 2 deliver to customers?

**Translating capabilities to value:**
| Capability Type | Value Translation |
|---|---|
| Speed/performance | Time saved, faster iteration, reduced waiting |
| Automation | Reduced manual work, fewer errors, consistency |
| Integration | Simplified workflow, eliminated context-switching |
| Transparency | Better decisions, reduced risk, auditability |
| Simplicity | Lower learning curve, faster onboarding, fewer mistakes |

**Rules:**
- Value must be stated from the customer's perspective, not the product's
- Include measurable outcomes where possible ("reduce X by Y%")
- Each value must map to a specific capability from Component 2

### 4. Target Customer Segments

**Question**: Who cares most about the value from Component 3?

**Segment definition template:**
```
Role: [specific job title or function]
Company: [size, stage, industry]
Trigger: [event that makes them search for a solution]
Must-have: [non-negotiable requirement]
```

**Rules:**
- Primary segment should be the smallest viable audience that gains the most value
- "Everyone" is not a segment — if you can't picture a single person, narrow further
- The trigger event is critical — without it, there's no urgency to buy
- Secondary segment is optional but useful for expansion planning

### 5. Market Category

**Question**: What frame of reference makes your product's value immediately obvious to the target segments?

**Category selection criteria:**
| Criterion | Good | Bad |
|---|---|---|
| Searchable | Buyers search for this term | Invented category nobody searches for |
| Expectation-setting | Sets helpful expectations about features | Misleads about what the product does |
| Competitive context | Competitors the customer already knows | No recognizable competitive context |

**Rules:**
- Use an existing market category — don't invent one
- Add a modifier to distinguish your approach within the category
- The positioning statement synthesizes all 5 components:
  `[Product] is a [category] that helps [segment] achieve [value] by [capability]`

## Evaluation Criteria

After completing all 5 components, evaluate the positioning:

| Criterion | Question | Pass |
|---|---|---|
| **Chain integrity** | Does each component build on the previous one? | Every claim traces back through the chain |
| **Specificity** | Could a competitor make the same claims? | No — positioning is unique to this product |
| **Customer voice** | Is value stated from the customer's perspective? | Yes — "you can..." not "we provide..." |
| **Falsifiability** | Could you prove each claim? | Yes — with data, demos, or testimony |
| **Simplicity** | Can you explain the positioning in 30 seconds? | Yes — the statement is clear and memorable |

## Common Mistakes

| Mistake | Problem | Fix |
|---|---|---|
| Starting with features | Inside-out positioning; ignores market context | Start with competitive alternatives |
| Skipping the status quo | Underestimates the "do nothing" competitor | Always include it in alternatives |
| Confusing features with capabilities | "We have a dashboard" vs "You can see X in real-time" | State what the customer can do, not what the product has |
| Category invention | "We created Intelligent DevOps Orchestration" | Use existing category + modifier |
| Universal targeting | "For all developers" | Narrow to who cares MOST about your specific value |
